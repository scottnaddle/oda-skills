# 성과지표 정의

성과 프레임워크의 각 수준(산출, 성과, 영향)에 대해 측정 가능하고 관리 가능한 지표를 상세 정의한다.

## 지표 정의 절차

### 1단계: 지표 초안 도출

로지컬 프레임워크 또는 성과 프레임워크의 각 성과 서술(Narrative)에서 지표를 도출한다. 하나의 성과 서술에 2~3개의 지표가 적절하다. 너무 많으면 데이터 수집 부담이 과도하고, 너무 적으면 성과를 다면적으로 파악하기 어렵다.

### 2단계: SMART 검증

모든 지표에 대해 5가지 기준을 체크한다:

- **S**pecific: "누가/무엇을/어디서" 측정하는가가 명확한가?
- **M**easurable: 수치 또는 등급으로 측정할 수 있는가?
- **A**chievable: 사업 기간과 예산 내에서 달성 가능한 수준인가?
- **R**elevant: 해당 성과를 적절히 반영하는가?
- **T**ime-bound: 측정 시점이 명시되어 있는가?

통과하지 못하는 기준이 있으면 지표를 수정한다.

### 3단계: 지표 시트(Indicator Reference Sheet) 작성

각 지표에 대해 상세 정의 시트를 작성한다:

```
┌─────────────────────────────────────────────────────┐
│ 지표명: [정확한 지표명]                               │
├─────────────────────────────────────────────────────┤
│ 정의: [지표가 무엇을 측정하는지 정확한 설명]           │
│ 산출식: [어떻게 계산하는지. 예: (A/B)×100]            │
│ 단위: [명, %, 건, 점 등]                             │
│ 성별분리: [Y/N — Y이면 남/여 별도 수집]               │
├─────────────────────────────────────────────────────┤
│ 기초선(Baseline): [값] (측정일: YYYY.MM)              │
│ 중간 목표(Mid-term Target): [값]                     │
│ 최종 목표(End Target): [값]                          │
├─────────────────────────────────────────────────────┤
│ 데이터 소스: [어디서 데이터를 얻는가]                  │
│ 수집 방법: [설문, 테스트, 관찰, 시스템로그 등]         │
│ 수집 도구: [설문지, 체크리스트, 추출쿼리 등]           │
│ 수집 빈도: [월/분기/반기/연]                          │
│ 수집 책임자: [직위/이름]                              │
├─────────────────────────────────────────────────────┤
│ 데이터 분석 방법: [기술통계, 비교분석, 추세분석 등]     │
│ 보고: [어떤 보고서에 포함되는가]                       │
│ 데이터 품질 관리: [검증 방법]                         │
└─────────────────────────────────────────────────────┘
```

### 4단계: 기초선 확보 전략

기초선이 없는 지표에 대해:
- **기존 데이터 활용**: 수원국 교육 통계, 이전 사업 자료, 행정 데이터
- **베이스라인 조사 실시**: 착수 단계에서 별도 조사 (baseline-survey-design 스킬 연계)
- **사업 초기 측정**: 연수 사전 테스트, 시스템 구축 전 현황 조사
- **대리 지표(Proxy) 활용**: 직접 측정이 어려우면 근사 지표 사용

### 5단계: 목표치 설정

목표치 설정의 근거:
- 유사 사업의 달성 수준 참조
- 수원국 국가 목표/전략의 목표치 참조
- 전문가 판단 (Expert Judgment)
- 기초선 대비 합리적 개선폭 (보통 20~50% 개선이 현실적)

목표치는 "도전적이되 달성 가능한(Stretch but Achievable)" 수준이어야 한다.

## 출력 형식

```
# 성과지표 정의서: [사업명]

## 지표 요약표
| # | 수준 | 지표명 | 기초선 | 목표치 | 빈도 |
|---|---|---|---|---|---|

## 지표별 상세 시트
### 지표 1: [지표명]
[Indicator Reference Sheet]

### 지표 2: [지표명]
[Indicator Reference Sheet]

## 기초선 확보 계획
[확보 필요 지표 + 방법 + 일정]

## 데이터 수집 일정 총괄
[연간 데이터 수집 캘린더]
```

## 흔한 실수

- 지표가 너무 많음 — 성과 수준당 2~3개면 충분. 20개 넘으면 관리 불가
- 기초선 없이 목표치만 설정 — "개선"을 입증할 수 없다
- 정성 지표를 정량화하지 않음 — 리커트 척도 등으로 수치화 필요
- 데이터 수집 책임자가 불명확 — "누군가 하겠지"는 아무도 안 한다는 뜻
