# 로지컬 프레임워크 작성

ODA 사업의 논리적 구조를 체계화하는 로지컬 프레임워크(Logframe/PDM)를 작성한다. 사업의 인과관계를 명확히 하고, 모니터링 및 평가의 기초를 제공한다.

## 왜 로지컬 프레임워크가 핵심인가

로지컬 프레임워크는 제안서의 뼈대다. 목표 체계가 흔들리면 방법론, 활동 계획, 예산, M&E 체계 모두 흔들린다. 평가위원은 로지컬 프레임워크의 논리적 일관성을 가장 먼저 본다. 인과관계가 부자연스러우면 아무리 화려한 방법론도 설득력이 떨어진다.

## PCM 로직 모델

상세 가이드 → `references/pcm-logframe-guide.md`

### 수직 논리 (Vertical Logic) — "If-Then" 체인

```
투입(Inputs) → 활동(Activities) → 산출물(Outputs) → 사업목적(Purpose/Outcome) → 상위목표(Goal/Impact)
```

각 수준 간의 "만약 ~하면, ~이 달성된다" 관계가 성립해야 한다:
- 만약 이 **투입**을 하면, 이 **활동**을 수행할 수 있다
- 만약 이 **활동**을 수행하면, 이 **산출물**이 생산된다
- 만약 이 **산출물**이 생산되면, 이 **사업목적**이 달성된다
- 만약 이 **사업목적**이 달성되면, 이 **상위목표**에 기여한다

각 수준 간 전환에는 **가정(Assumptions)**이 있다 — 외부 환경이 특정 조건을 충족해야 다음 수준으로 이어진다.

### 수평 논리 (Horizontal Logic)

각 수준별로:
- **서술(Narrative Summary)**: 무엇을 달성하려는가
- **OVI(Objectively Verifiable Indicators)**: 어떻게 측정하는가
- **MoV(Means of Verification)**: 데이터를 어디서 얻는가
- **Assumptions**: 이 수준이 다음 수준으로 이어지려면 어떤 외부 조건이 필요한가

## 작성 절차

### 1단계: 문제 분석 (Problem Analysis)

사업이 해결하려는 문제를 구조화한다:
- **핵심 문제(Core Problem)** 정의
- **원인 분석**: 핵심 문제의 원인을 2~3개 수준으로 분해
- **결과 분석**: 핵심 문제가 해결되지 않을 때의 부정적 결과
- 문제 나무(Problem Tree) 구성이 유용

### 2단계: 목표 분석 (Objective Analysis)

문제 나무를 뒤집어 목표 나무(Objective Tree)를 구성한다:
- 문제 → 목표로 전환 (부정문 → 긍정문)
- 원인 → 산출물/활동으로 전환
- 결과 → 상위목표/사업목적으로 전환

### 3단계: PDM 매트릭스 작성

4×4 매트릭스를 채운다:

```
| 구분 | 서술 (Narrative) | OVI (지표) | MoV (검증수단) | 가정 |
|---|---|---|---|---|
| 상위목표 | [Goal] | [지표] | [데이터 소스] | - |
| 사업목적 | [Purpose] | [지표] | [데이터 소스] | [가정] |
| 산출물 1 | [Output 1] | [지표] | [데이터 소스] | [가정] |
| 산출물 2 | [Output 2] | [지표] | [데이터 소스] | [가정] |
| 산출물 3 | [Output 3] | [지표] | [데이터 소스] | [가정] |
| 활동 | [Activities] | [투입(Inputs)] | [예산] | [전제조건] |
```

### 4단계: 지표 품질 검증

모든 OVI에 대해 SMART 기준을 검증한다:
- **S**pecific: 무엇을, 누구를, 어디서 측정하는지 구체적인가?
- **M**easurable: 정량적으로 또는 체계적으로 측정 가능한가?
- **A**chievable: 사업 기간 내 달성 가능한 수준인가?
- **R**elevant: 해당 목표/산출물의 달성을 적절히 반영하는가?
- **T**ime-bound: 측정 시기가 명시되어 있는가?

### 5단계: 가정(Assumptions) 검증

각 가정에 대해:
- 발생 가능성은 어느 정도인가? (높으면 리스크로 관리)
- 사업 통제 범위 내인가 밖인가?
- 가정이 충족되지 않으면 어떤 대안이 있는가?

## 출력 형식

```
# 로지컬 프레임워크: [사업명]

## 1. 문제 분석
### 핵심 문제
[서술]
### 문제 나무
[원인-문제-결과 구조]

## 2. PDM (Project Design Matrix)
[4×4 매트릭스 표]

## 3. 지표 상세
[지표별 정의, 측정방법, 빈도, 책임, 기초선, 목표치]

## 4. 가정 분석
[가정별 발생확률, 영향도, 대안]

## 5. 성과 체계도
[시각적 인과관계 다이어그램]
```

## 흔한 실수

- 산출물(Output)과 활동(Activity)을 혼동 — "워크숍 실시"는 활동이지 산출물이 아니다. "교원 200명의 ICT 활용 역량 향상"이 산출물이다
- 지표가 너무 추상적 — "교육 품질 향상"이 아니라 "교원의 ICT 활용 수업 비율 30%에서 70%로 증가"
- 가정을 빠뜨림 — 모든 인과관계에는 외부 조건이 있다
- 수직 논리의 단절 — 산출물이 모두 달성되어도 사업목적과 연결이 안 되는 경우
